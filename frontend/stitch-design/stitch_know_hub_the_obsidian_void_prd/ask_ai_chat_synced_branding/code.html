Summarize the findings from the Obsidian project regarding latency issues during high-volume ingestion. Target specific bottlenecks mentioned in the engineering logs.

Based on the engineering logs for the Obsidian project, high-volume ingestion latency is primarily localized to two critical bottlenecks within the pipeline:

1. Vector Embedding Queue Congestion
During batch uploads exceeding 10,000 documents, the primary embedding service experiences significant thread starvation. The logs indicate a wait time increase from baseline 45ms to over 850ms per batch [1]

sys_log_2023_10_14.txt
"CRITICAL: Embedding queue length exceeded 5000. Thread pool exhausted. Latency spike to 852ms detected in batch-processor-04."
. This is exacerbated when chunking dense PDF documents with complex tables.

2. Index Write Locks
The secondary bottleneck occurs at the database layer. The vector index requires periodic write-locks during rebalancing operations. When ingestion volume spikes concurrently with these rebalance events, write queries are queued, causing a cascading delay back up to the API layer [2]

db_metrics_summary.md
"Rebalancing operation (ID: RB-992) forced write lock for 4.2s. 14,000 incoming vectors queued. API timeout threshold nearly breached."
.

RECOMMENDATION: Engineering suggests implementing a secondary queue for embedding tasks and decoupling the vector write operations into asynchronous micro-batches.
